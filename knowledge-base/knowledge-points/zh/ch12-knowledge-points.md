# 第12章：自动化数据收集方法 — 知识点

## 1. 核心概念定义

**自动化数据收集 (Automated Data Collection)** 指使用软件工具收集大量用户交互数据，而研究人员几乎无需直接努力。这些方法范围从现有商品软件（Web 日志）到系统观察工具（活动记录器）再到定制研究软件（仪表化应用程序）。这些方法在易用性与灵活性之间存在一个谱系。

**Web 日志 (Web Logs)** 是由 Web 服务器、电子邮件服务器和数据库服务器生成的文件，用于存储请求和活动记录。它们提供了对服务器发出的所有请求的顺序列表，包括主机 IP 地址、时间戳、HTTP 方法、请求的资源、协议版本、状态代码、大小、引用者和用户代理。Web 日志是 HCI 研究的有力工具，能够进行可用性评估、使用模式分析和经验性任务性能研究。

**通用日志格式 (Common Log Format)** 是大多数 Web 服务器用于格式化日志文件的标准格式。它包括主机地址、时间戳、HTTP 请求详细信息、状态代码和大小等字段。Web 服务器可以自定义以包含额外字段或为特定研究需求创建专用日志文件。

**Web 代理 (Web Proxy)** 是一个中间服务器，接收来自一组用户的所有 Web 请求，检索请求的材料并将其返回给用户。与仅捕获对单个站点请求的标准 Web 服务器日志不同，Web 代理捕获来自所有用户跨多个网站的所有请求。Web 代理可以拦截和修改用户请求和服务器响应。

**存储的应用程序数据 (Stored Application Data)** 指计算机使用留下的痕迹，提供有关用户如何与应用程序交互和管理信息的示例。示例包括文件系统结构、GUI 桌面安排、电子邮件文件夹组织、Web 书签和社交网络连接。这些数据由用户日常使用的工具自动收集。

**活动记录软件 (Activity-Logging Software)** 在后台不可见地运行，记录鼠标移动、键盘输入和窗口系统交互，包括窗口移动、调整大小、打开和关闭。这些工具充当用户交互事件的代理，在事件发生时记录它们，然后将它们传递给应用程序或操作系统环境。

**键盘记录器 (Keystroke Logger)** 是活动记录软件的一个特殊子类，仅关注键盘输入。虽然对研究有价值，但当这些工具被用于秘密记录用户交互时，它们作为“间谍软件”而声名狼藉。

**Web 代理（研究导向）(Web Proxy - Research-Oriented)** 与缓存代理不同，它充当研究中介。代理捕获所有用户的完整会话数据，提供比标准服务器日志更广泛的用户活动图景。像 WebQuilt 这样的研究代理将日志记录与可视化和分析功能相结合。

**仪表化软件 (Instrumented Software)** 指添加了测量和记录工具以跟踪使用情况的应用程序。构建仪表化软件需要修改源代码以处理用户交互并记录它们。方法包括使用商业产品中的宏/扩展设施、修改开源软件或在供应商合作下向商业产品添加仪表化。

**交互记录工具 (Interaction Recording Tools)** 捕获屏幕视频和潜在的麦克风音频，提供发生了什么以及何时发生的记录。它们范围从商品屏幕录制器到专用的可用性研究包，如 TechSmith 的 Morae，后者通过鼠标/键盘记录和网络摄像头集成增强了基本捕获。

**定制研究软件 (Custom Research Software)** 是专门为运行实验而创建的软件，向用户呈现任务并记录完成时间、错误和其他相关数据。示例包括测量鼠标移动准确性的菲茨定律 (Fitts' Law) 研究。

**粒度 (Granularity)** 指收集数据中的细节量。细粒度、高分辨率数据涉及每个可能的用户交互事件（单个按键、鼠标移动）。粗粒度、低分辨率数据包含较少的事件（菜单选择、按钮点击、对话交互）。适当的粒度取决于研究问题。

**混合数据收集 (Hybrid Data Collection)** 结合多个数据收集渠道以克服任何单一方法的缺点。示例包括将日志文件与视频记录相结合、代理与仪表化浏览器相结合，或将自动化捕获与人工观察相结合。软件工具可以同步这些数据流以进行集成分析。

**解析 (Parsing)** 是逐行读取日志文件、将每个条目分解为其组成部分并更新保持不同事件类型计数和统计数据的数据结构的过程。解析器通常用 Perl 和 Python 等脚本语言编写。

**菲茨定律 (Fitts' Law)** 指出，在目标之间移动所需的时间是移动幅度与目标宽度之比的对数的函数。随着目标之间距离的增加或目标尺寸的减小，移动时间增加。这对界面设计有深远的影响。

**数据挖掘 (Data Mining)** 涉及使用关联规则和聚类等技术从大型数据集中提取模式。在 HCI 中，数据挖掘可以识别经常共同发生的用户操作，根据使用模式将用户聚类为组，并预测购买等结果。

## 2. 理论框架

**易用性与灵活性谱系 (Ease-of-Use vs. Flexibility Spectrum)**：自动化数据收集方法存在于一个从易于使用但有限（现有软件工具）到灵活但劳动密集（定制软件）的谱系上。Web 日志最简单，但收集的数据可能有限。活动记录器更强大，但需要配置。定制软件可以满足精确的研究需求，但需要大量的开发工作。

**三类研究问题 (Three Categories of Research Questions)**：自动化数据收集解决三大类问题：(1) 对信息管理行为的回顾性分析（检查计算机使用的痕迹），(2) 受控实验（使用 Web 日志或定制软件收集计时和相关数据），以及 (3) 可用性研究和用户如何使用工具的探索。

**粒度-分析权衡 (Granularity-Analysis Trade-off)**：收集的数据量（粒度）必须与研究问题相匹配。细粒度数据（每次鼠标移动）对于长期使用研究来说是压倒性的。粗粒度数据（应用程序级使用）不足以理解特定的交互模式。理想情况下，工具应允许灵活的粒度，并按级别标记事件。

**代理与直接仪表化 (Proxies vs. Direct Instrumentation)**：代理透明地拦截交互而不修改应用程序，提供广泛的覆盖范围但细节有限。直接仪表化修改应用程序以捕获特定事件，提供精确的数据但需要访问源代码。每种方法在覆盖范围、精确度和实施工作量方面都有权衡。

**自动化检查模型 (Automated Inspection Model)**：自动化检查工具分析界面布局、内容和语言以确定指南合规性。虽然对识别明显问题有用，但这些工具应补充而不是取代传统的设计评审和用户测试。它们的效用受底层指南的有效性和范围的限制。

**自动化收集中的生态效度 (Ecological Validity in Automated Collection)**：自动化方法可以在自然环境中以最少的研究人员干扰收集数据，与实验室观察相比，可能增加生态效度。然而，缺乏关于用户目标的背景知识意味着仅日志数据可能不足以理解观察到的行为背后的“为什么”。

## 3. 方法论步骤

### 步骤1：确定研究问题和数据需求

在收集数据之前，精确确定您希望了解什么以及收集的数据将如何帮助回答这些问题。确定适当的粒度级别。考虑您是否需要单个按键、菜单选择、应用程序级使用或长期使用模式。收集太多、太少或错误的数据会浪费资源。

### 步骤2：选择适当的数据收集工具

根据研究需求在易用性/灵活性谱系上选择工具：
- **现有软件**：当默认数据收集足够时使用（Web 日志、存储的应用程序数据）
- **活动记录器**：当您需要跨应用程序的更广泛交互跟踪时使用
- **Web 代理**：当您需要跟踪用户跨多个网站的活动时使用
- **仪表化软件**：当您需要现有工具未捕获的特定数据时使用
- **定制软件**：当您需要完全控制数据收集时使用

### 步骤3：配置数据收集

对于 Web 日志：配置服务器以包含必要的字段，考虑创建专用日志，对公共站点使用机器人排除协议，并在实验期间禁用浏览器缓存。对于代理：确保足够的处理能力和网络带宽，配置浏览器设置或路由器级代理。对于仪表化软件：修改源代码以记录感兴趣的事件，将记录的操作与类别相关联以便于分析。对于所有方法：每个会话以空日志文件开始，将完成的会话日志移动到单独的目录。

### 步骤4：解决隐私和伦理问题

在收集数据之前实施隐私保护。考虑匿名化 IP 地址、从日志中删除识别信息、将数据收集限制在严格必要的范围内，并在分析完成后销毁数据。确保网站有关于数据收集的隐私政策。对于基于代理的研究，要注意用户可能忘记他们的请求正在被记录。检查是否需要机构审查委员会 (IRB) 批准。

### 步骤5：进行数据收集

在适当控制下运行研究。对于使用 Web 日志的经验研究：将任务作为 Web 链接呈现，使用本地服务器以最小化网络延迟，通过链接选择记录开始和结束事件。对于活动记录研究：确保记录软件正确配置，验证反间谍软件工具不会干扰，向参与者提供清晰的同意书。

### 步骤6：解析和分析日志文件

编写或使用现有的解析器（通常用 Perl 或 Python）逐行读取日志文件，提取相关字段并计算统计数据。使用既定的 Web 日志分析工具（公共站点使用 Google Analytics，或研究使用专用工具）。对于非 Web 应用程序，考虑将日志格式化为与 Web 服务器格式匹配以实现工具兼容性。使用可视化工具（如 WebQuilt）将使用模式显示为图形。

### 步骤7：应用高级分析技术

对于复杂问题，采用序列分析来研究事件顺序，数据挖掘来提取模式，以及机器学习来对用户会话进行分类。使用交互式工具在多个粒度级别可视化数据。考虑将自动化数据与定性观察相结合以获得更丰富的理解。

### 步骤8：管理数据存储

在日志文件（简单的文本文件，易于写入）和数据库（使用 SQL 进行强大查询，当应用程序已经使用数据库时有用）之间选择。设计数据存储以促进分析。对于数据库驱动的 Web 应用程序，添加用户活动表是直接的。

## 4. 决策标准

**何时使用 Web 日志**：用于研究基于 Web 的交互、站点可用性评估和涉及 Web 链接选择的经验研究。Web 日志是理解访问了哪些页面、用户如何在页面之间导航以及交互何时发生的理想选择。限制包括缺乏关于用户目标的背景知识以及无法跟踪客户端 JavaScript 交互。

**何时使用 Web 代理**：当您需要跟踪用户跨多个网站（而不仅仅是单个站点）的活动时使用。代理提供完整的会话数据，但需要大量的技术设置、处理能力和带宽。考虑用于一般浏览习惯和多站点使用模式的研究。

**何时使用活动记录器**：用于研究多应用程序工作流、应用程序切换模式和长期使用研究。活动记录器在操作系统级别捕获交互，提供比应用程序特定工具更广泛的覆盖范围。仔细考虑隐私影响。

**何时使用仪表化软件**：当现有工具未捕获您需要的特定数据时使用。对于研究复杂应用程序使用模式特别有价值。考虑修改所需的工作量以及宏/扩展设施或开源代码是否可用。

**何时使用定制软件**：用于需要精确控制任务呈现和数据收集的结构化实验。最适合菲茨定律研究、菜单比较实验和其他受控调查。

**粒度选择**：将数据粒度与研究问题相匹配：
- 长期使用模式：粗粒度（应用程序级、功能级）
- 任务完成研究：中粒度（菜单选择、按钮点击）
- 详细交互分析：细粒度（按键、鼠标移动）
- 考虑支持具有事件标记的多粒度级别的工具

**混合与单一方法**：当单一方法无法提供足够的数据时，使用混合方法。将日志数据与视频相结合以获得上下文理解，代理与仪表化浏览器相结合以获得全面覆盖，或自动化捕获与人工观察相结合以获得更丰富的解释。

## 5. 书本案例与示例

**顺序菜单与同时菜单研究**：对访问美国人口普查局经济数据的 Web 菜单结构进行比较。顺序菜单允许用户选择县→业务类别→年份，而同时菜单允许以任何顺序选择。使用本地托管的 Web 服务器和标准日志文件，研究人员发现顺序菜单在简单任务上表现良好，但对于更复杂的任务，同时菜单更可取。任务完成时间从日志时间戳中提取。

**WebQuilt**：一个专门为可用性数据收集设计的代理服务器。WebQuilt 将日志记录与将日志条目转换为推断的用户操作、聚合到图形结构以及可视化组件相结合。与通用代理不同，WebQuilt 处理特定于站点的数据收集，无需浏览器配置，通过修改页面将请求路由通过代理。

**MouseTracks 和 UsaProxy**：基于 JavaScript 的系统，通过记录低级交互数据（包括鼠标移动）的代码修改网页。这些数据被发送到服务器进行记录和可视化，提供比标准代理更细粒度的数据。

**Microsoft Office 2003 仪表化**：微软的客户体验改善计划从超过 130 亿个用户会话中收集匿名使用数据。数据包括菜单选择、键盘快捷键和自定义设置。主要发现：粘贴命令是最受欢迎的（占所有 Word 命令使用的 11%），粘贴按钮是最常点击的工具栏按钮。这为 Office 2007 界面的重新设计提供了信息。还证实了前五个 Word 命令占所有使用的 32% 以上。

**个性化界面研究（McGrenere 等人）**：使用 Microsoft Word 2000 中的脚本工具构建了一个扩展，允许用户使用简化的个性化界面。对 20 名用户进行的实地研究安装了捕获使用直方图的记录工具。结果表明，只有少数命令被非常频繁地使用，用户将几乎所有的这些命令都添加到了他们的个性化界面中。

**Ingimp（仪表化 GIMP）**：GNU 图像处理程序的仪表化版本，收集使用计时、窗口/图层数量、命令使用和任务切换详细信息。隐私措施包括不记录特定的按键或鼠标位置、提供启动对话框以禁用记录，以及使所有源代码可供检查。

**菲茨定律儿童鼠标研究**：定制软件跟踪了 13 名四岁儿童、13 名五岁儿童和 13 名成人的鼠标移动准确性。参与者在不同大小（16、32、64 像素）和不同距离（128、256、512 像素）的目标之间移动。儿童较慢、较不准确，并表现出更多的目标悬停和重新进入。鼠标运动路径显示儿童移动范围广泛，而成人移动准确。

**DART（中断和恢复跟踪器）**：记录窗口位置、大小、操作和警报，同时将键盘记录限制为菜单快捷键和标点符号。对 27 人 2200 多小时的分析产生了关于因电子邮件/即时消息警报损失的时间以及用户恢复模式的见解。示范了具有隐私保护的负责任的数据收集。

**LAB-IN-A-BOX**：结合了定向音频、眼动追踪、屏幕捕获、鼠标移动和 Kinect 身体方向数据，用于研究患者就诊期间的电子病历。配置在滚动箱中，可在 10 分钟内设置。使用 ChronoViz 进行同步多流分析，被描述为“计算民族志”。

## 6. 常见错误与陷阱

**收集太多数据**：收集一切“以防万一”的诱惑导致压倒性的数据量，难以分析。即使存储成本低廉，筛选数兆字节的事件日志以寻找有意义的模式也具有挑战性。围绕特定研究问题设计数据收集。

**收集太少数据**：相反，收集不足的数据或粒度错误的数据可能无法回答研究问题。详细交互研究需要细粒度数据；粗粒度数据足以满足长期模式。

**忽视隐私影响**：活动记录器可以捕获敏感信息，包括文档标题、电子邮件地址和 Web 浏览习惯。Web 代理跟踪所有请求，包括可能令人尴尬的站点。IP 地址可以识别特定计算机并可能识别个人。始终实施隐私保护，获得知情同意，并考虑机构审查委员会 (IRB) 批准。

**未考虑浏览器缓存**：浏览器缓存存储最近访问页面的本地副本。对缓存页面的请求不会生成服务器日志条目，可能导致数据不完整。在受控实验期间禁用缓存。

**在日志分析中混淆相关性与因果关系**：页面请求之间的长间隔可能表示仔细阅读或分心。在帮助页面和搜索页面之间反复循环可能表示任务失败或有意识的探索。日志数据缺乏关于用户目标的背景知识；与观察或访谈相结合。

**Web 日志的机器人污染**：可公开访问的网站接收来自 Web 机器人（搜索引擎爬虫）的请求。这些自动化工具跟踪链接并检索页面，污染日志数据。在数据收集期间短时间内使用机器人排除协议。

**忽视会话识别挑战**：Web 日志可能包含来自多个用户的交错请求。防火墙使来自不同用户的请求看起来来自同一台机器。从日志文件识别单个用户会话是不完美的。Cookie 可能被禁用。浏览器配置可能提供误导性的用户代理和引用者信息。

**代理服务器容量不足**：如果代理服务器无法处理处理和带宽需求，用户将经历浏览延迟，可能改变他们的习惯或拒绝使用代理。这会使数据收集产生偏见。确保专用、充足的资源。

**未验证自动化数据收集**：未能在全面部署之前对数据收集和分析工具进行试点测试可能导致收集无法使用的数据。验证收集的数据确实告诉您您想要知道的内容。

**过度依赖自动化工具**：用于界面评估的自动化检查工具应补充而不是取代传统的设计评审和用户测试。它们的效用受底层指南有效性的限制。

**未记录仪表化决策**：在仪表化软件时，未能记录记录了哪些事件、以什么粒度以及有什么隐私保护，使得难以解释结果或复制研究。

**忽视高级任务推断挑战**：低级交互事件（按键、鼠标点击）不会直接揭示高级任务（格式化文本、插入表格）。从交互数据推断任务需要领域知识和模式识别，这是不完美的。具有相同功能的多条路径的应用程序使分析复杂化。

**未将自动化数据与定性上下文相结合**：日志文件提供了发生了什么以及何时发生的详细记录，但缺乏关于用户为何如此行事的信息。将自动化数据与观察、访谈或有声思维协议相结合提供了更丰富的理解。
